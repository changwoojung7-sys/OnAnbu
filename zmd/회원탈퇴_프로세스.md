# On-Anbu 회원 탈퇴 프로세스 설계 및 구현 (최종)

작성일자: 2026-03-05
최종 수정: 2026-03-05 (Edge Function v4 반영)

## 1. 개요

On-Anbu 앱은 "부모님(케어대상)"과 "보호자(주/보조케어자)" 간의 강력한 유대감을 바탕으로 합니다. 따라서 한쪽의 탈퇴가 다른 쪽의 소중한 추억(기록) 파기로 이어질 수 있어, 역할에 따른 차등화된 탈퇴 로직과 명확한 데이터 파기 권한을 부여하도록 설계되었습니다.

---

## 2. 부모님(Care Recipient) 탈퇴 로직

- **위치**: `app/parent/settings.tsx`
- **구동**: Edge Function (`withdraw-parent`) 호출

### 2.1 다단계 확인 UI (Web 호환성 강화)

웹 환경의 브라우저 제약을 고려하여 모달 내에서 단계별로 선택할 수 있는 **State 기반 확인 로직**을 적용했습니다.

#### 옵션 1. 내 계정만 탈퇴 (기록 유지 - Soft Delete)

- **목적**: 앱 이용만 중단하고, 내가 남긴 안부 기록들은 자녀(보호자)들이 계속 볼 수 있도록 보존하고 싶은 경우.
- **로직**:
  1. **개인정보 삭제**: 본인의 프로필 사진(Avatar)을 Storage에서 즉시 삭제합니다.
  2. **초대 무효화**: 진행 중인 `parent_invitations` 상태를 `cancelled`로 변경하여 동일 초대코드로 재접속을 차단합니다.
  3. **계정 비활성화**: `auth.users`에서 이메일을 난수화(`uuid@withdrawn.local`)하여 실제 계정은 보존하되 접근은 불가능하게 처리합니다. (기존 로그 데이터 보존 ✅)

#### 옵션 2. 모든 기록 삭제 후 탈퇴 (Full Delete - Hard Delete)

- **목적**: 본인이 전송한 모든 사진, 메시지 등 활동 내역을 완전히 파기하고 싶은 경우.
- **로직**:
  1. **데이터 완전 파기**: 아바타 및 본인이 작성한 모든 안부 기록(`action_logs`)에 연동된 미디어 파일을 Storage에서 영구 삭제합니다.
  2. **초대 무효화**: 초대장 상태를 `cancelled`로 변경합니다.
  3. **계정 영구 삭제**: `auth.users`에서 계정을 삭제합니다. CASCADE 설정에 의해 DB 내 모든 활동 로그가 물리적으로 삭제됩니다.

## 3. 케어자(Guardian) 탈퇴 로직

- **위치**: `app/settings.tsx`
- **구동**: Edge Function (`withdraw-guardian`) 호출

### 3.1 다단계 확인 UI (Web 호환성 강화)

웹 환경에서 브라우저 알림창(`Alert`)이 차단되거나 응답하지 않는 문제를 해결하기 위해, 모달 내부에서 화면이 전환되는 **State 기반 다단계 확인 로직**을 적용했습니다.

### 3.2 탈퇴 옵션 상세

#### 옵션 1. 내 계정만 탈퇴 (기록 유지 - Soft Delete)

- **목적**: 앱 이용만 중단하고, 내가 남긴 기록은 가족들이 계속 볼 수 있게 함.
- **로직**: `auth.users`의 이메일을 난수화(`uuid@withdrawn.local`)하고 패스워드를 제거하여 계정만 비활성화합니다.

#### 옵션 . 모든 가족과 앱 이용 중단 및 내역 삭제 (Hard Delete)

- **목적**: 가족 그룹의 생성자(주케어자)로서 모든 데이터를 파기하고 싶은 경우.
- **로직**:
  1. 본인이 초대한 모든 부모님(`parent_invitations` 기준)의 계정 ID 수집.
  2. 본인 및 부모님들의 모든 아바타와 그룹 내 모든 활동 로그의 미디어 파일을 Storage에서 일괄 삭제.
  3. 모든 연결된 부모 계정 및 본인 계정을 `auth.users`에서 삭제하여 전체 그룹 정보와 로그를 물리적으로 파기합니다.

## 4. 기술적 특징 및 보안

- **Edge Functions**: 관리자 권한(`service_role`)이 필요한 계정 삭제 및 Storage 조작을 위해 클라이언트가 아닌 서버 측에서 로직을 안전하게 처리합니다.
- **JWT 수동 검증**: 서용 환경에 따른 401 오류를 방지하기 위해 함수 내부에서 `auth.getUser()`를 통한 수동 토큰 검증 로직을 도입했습니다.
- **CORS 및 Web 대응**: 크로스 오리진 요청 및 웹 환경의 UI 제약을 고려하여 전용 헤더 및 컴포넌트 구조를 최적화했습니다.

## 5. 결론

이 프로세스를 통해 온안부는 부모님의 우발적인 접근 차단을 보장하면서도, 주케어자에게는 데이터 보존과 완전 파기라는 명확한 선택권을 제공하여 사용자 신뢰를 높였습니다.
