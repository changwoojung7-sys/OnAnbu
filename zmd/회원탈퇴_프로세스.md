# On-Anbu 회원 탈퇴 프로세스 설계 및 구현

작성일자: 2026-03-05
버전: 1.0.0

## 1. 개요

On-Anbu 앱은 "부모님(케어대상)"과 다수의 "케어자(주/보조케어자)"가 가족 그룹을 맺고 사진, 음성, 안부 등의 미디어를 주고받는 형태입니다.
따라서 회원 탈퇴 시 한 명의 이탈이 다른 가족 구성원들의 데이터 열람에 치명적인 영향을 줄 수 있습니다. 이를 방지하기 위해 각 역할(부모님, 주 케어자)에 따라 다른 **Soft Delete(계정 난수화)** 및 **Multi-step (다단계) 옵션형 탈퇴** 로직을 도입했습니다.

---

## 2. 부모님(Parent) 탈퇴 로직

부모님 계정(`app/parent/settings.tsx`)은 로그인 수단이 없고 초대 코드로만 진입합니다.

### 2.1 요구사항

- 부모님이 탈퇴하여도 해당 부모님 앞으로 쌓인 기존 가족 기록, 사진, 음성 안부는 주케어자가 여전히 볼 수 있어야 한다. (기록 보존)
- 하지만 탈퇴하는 즉시 부모님의 기기에서는 계정이 로그아웃 되어야 하며, 같은 초대코드로 재접속할 수 없어야 한다.
- 이후 주케어자가 앱에서 새로운 초대코드를 발급하면, 새로운 계정으로 다시 들어올 수 있어야 한다.

### 2.2 구현 방식 (Soft Delete 연동 RPC: `withdraw_parent`)

- **[취소]** `parent_invitations` 테이블에서 부모님이 수락했던 현재 초대장의 상태를 `'cancelled'`로 업데이트합니다.
- **[난수화]** `auth.users` 테이블에서 이메일을 `UUID@withdrawn.local` 등 무작위 문자열로 변경하고, `encrypted_password`와 `raw_user_meta_data`를 비웁니다.
- **[로그아웃 처리]** `profiles` 테이블에서 푸시 토큰(`push_token`)을 비우고, 프론트엔드에서는 클라이언트 로그아웃을 실행하여 첫 로그인 화면으로 되돌립니다.
- 결과적으로 `DELETE` 쿼리가 발생하지 않으므로(Hard Delete 방지), `action_logs`와 `family_groups` 데이터의 외래키(Foreign Key) CASCADE 영향을 받지 않고 데이터가 보존됩니다.

## 3. 케어자(Guardian) 탈퇴 로직

주 케어자(`app/settings.tsx`) 계정에는 일반적인 "탈퇴"에 그치지 않고, 그가 생성한 가족 그룹 전체 데이터를 어떻게 할 것인지 책임을 묻는 과정이 필요합니다.

### 3.1 요구사항

- 설정 메뉴 내 "회원탈퇴" 탭 진입 시, **다단계 모달 창**을 표시합니다. 옵션에 따라 본인만 탈퇴할지, 모든 얽힌 가족들의 데이터를 파기할지 결정합니다.

#### 옵션 1. 내 계정만 탈퇴 (기록 유지)

- **대상**: 본인 계정만 앱 이용을 멈추거나 보조케어자가 있는 경우.
- **상세**: `auth.users`만 난수화(Soft Delete)시켜 기존 이메일 주소를 릴리즈하고 `profiles` 테이블은 남겨둡니다.

#### 옵션 3. 가족 전체 이용 중단 및 데이터 완전 파기

- **대상**: 더 이상 온안부 서비스를 나뿐만 아니라 내가 초대한 부모님들도 이용하지 않고 데이터를 모두 삭제하고 싶을 때.
- **상세**: (다이얼로그에서 경고성 메시지 표출 후 진행)
  본인이 발급한 `parent_invitations`을 바탕으로 초대된 **부모 계정(`auth.users`)들을 찾아서 `DELETE` (Hard Delete)**.
  그다음 **본인 계정(`auth.users`)도 `DELETE`**.
  Supabase에 설정된 Foreign key `ON DELETE CASCADE` 룰에 의해 관련된 `family_groups`, `action_logs`(미디어/안부 정보) 등이 모두 **물리적으로 삭제**됩니다.

### 3.2 구현 방식 (옵션형 분기 RPC: `withdraw_guardian(p_option INT)`)

- 프론트엔드(`app/settings.tsx`)에서 사용자가 선택한 옵션값(`1` 또는 `3`)을 RPC로 넘깁니다.
- Option 1 (Soft Delete):
  `UPDATE auth.users SET email = UUID, encrypted_password = '' WHERE id = auth.uid()`
- Option 3 (Hard Delete Cascade):
  부모 계정(`accepted_by`) 추출 -> 부모 계정들 `DELETE FROM auth.users` -> 연쇄 삭제 발동 -> 주 케어자 로그인 계정 `DELETE FROM auth.users`

## 4. UI/UX 구성 흐름 (다이얼로그)

1. **내 정보 > 설정 (`app/settings.tsx`)**
   - 하단 [로그아웃] 오른쪽에 [회원탈퇴] 버튼 배치 제공 (잘 안 보이되 누르기 쉽게 인접 배치).
2. **모달 (BottomSheet / Dialog) 등장**
   - 상단 텍스트: "계정 복구가 불가능합니다. 탈퇴 방식을 선택해주세요."
   - 버튼 1: `[내 계정만 탈퇴 (나머지 가족/기록 유지)]` (회색 배경)
   - 버튼 2: `[모든 가족과 앱 이용 중단 및 내역 삭제]` (강렬한 붉은색)
3. **재확인 경고 (Alert)**
   - 버튼 2번(완전 삭제)을 누르면 `Alert.alert`를 띄웁니다.
   - 메시지: "가족들과 조율하셨습니까? 회원님이 초대한 부모님 계정들을 비롯한 모든 활동, 안부, 기록 등이 영구 삭제됩니다."
   - `[취소]`와 `[삭제 및 탈퇴]` 버튼 중 선택.

---

## 5. 결론 및 보완

이 프로세스를 통해 온안부는 부모의 우발적인 탈퇴(앱 삭제/정보 초기화)에도 자녀(보호자)가 과거의 추억(미디어)을 잃지 않도록 보완되었으며, 메인 관리자인 주 케어자가 탈퇴 시에도 강제 삭제 우려 없이 선택지를 쥐고 안전히 회원 탈퇴를 진행할 수 있는 로직이 적용되었습니다.
